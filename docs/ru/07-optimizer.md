# Глава 7: Оптимизатор (BYOM)

*Аудитория: медиабайеры, инженеры по оптимизации*

Оптимизатор — это автоматический движок оптимизации Cat-Scan. «BYOM» означает
Bring Your Own Model: вы регистрируете внешний эндпоинт скоринга, а Cat-Scan
использует его для генерации предложений по изменению конфигураций.

## Как это работает

```
  Score          Propose          Review          Apply
────────────> ────────────> ────────────> ────────────>
Your model     Cat-Scan       You (human)    Google AB
evaluates      generates      approve or     config is
segments       config         reject         updated
               changes
```

1. **Score**: Cat-Scan отправляет данные по сегментам на эндпоинт вашей модели.
   Модель возвращает оценку для каждого сегмента (гео, размер, паблишер).
2. **Propose**: на основе оценок Cat-Scan генерирует конкретные изменения
   претаргетинга (например, «исключить эти 5 гео», «добавить эти 3 размера»).
3. **Review**: вы видите предложение с прогнозируемым эффектом. Вы одобряете или
   отклоняете.
4. **Apply**: одобренные предложения применяются к конфигурации претаргетинга
   на стороне Google. Изменение фиксируется в истории.

## Управление моделями

### Регистрация модели

Перейдите в `/settings/system` и найдите раздел «Оптимизатор».

1. Нажмите **Register Model**.
2. Заполните: название, тип модели, URL эндпоинта (ваш сервис скоринга).
3. Эндпоинт должен принимать POST-запросы с данными сегментов и возвращать
   результаты с оценками.
4. Сохраните.

### Проверка эндпоинта

Перед активацией протестируйте модель:

1. Нажмите **Validate endpoint** на карточке модели.
2. Cat-Scan отправит тестовый запрос на ваш эндпоинт.
3. Результаты покажут: время отклика, корректность формата ответа, распределение
   оценок.
4. Устраните все проблемы перед активацией.

### Активация и деактивация

- **Activate**: модель становится активным скоринговым движком для данного сита.
- **Deactivate**: модель перестаёт использоваться, но её конфигурация сохраняется.
  Одновременно на один сит может быть активна только одна модель.

## Пресеты рабочего процесса

При запуске цикла «оценка + предложение» вы выбираете пресет:

| Пресет | Поведение | Когда использовать |
|--------|----------|-------------------|
| **Safe** | Предлагает изменения только с высокой уверенностью и низким риском. Небольшие улучшения, меньшая вероятность ошибок. | Первое использование оптимизатора или консервативные аккаунты. |
| **Balanced** | Умеренный порог уверенности. Хороший баланс между эффектом и безопасностью. | Режим по умолчанию для большинства случаев. |
| **Aggressive** | Предлагает более масштабные изменения с большим потенциальным эффектом. Выше риск чрезмерной оптимизации. | Опытные пользователи, которые следят за показателями ежедневно и могут быстро откатить изменения. |

## Экономика

Оптимизатор также отслеживает экономику оптимизации:

- **Effective CPM**: фактическая стоимость за тысячу показов с учётом потерь.
- **Hosting cost baseline**: стоимость инфраструктуры вашего бидера, указанная
  в настройках оптимизатора. Используется для расчёта того, покрывает ли экономия
  от снижения QPS затраты на хостинг.
- **Efficiency summary**: общее соотношение полезного QPS к общему QPS.

Настройте стоимость хостинга в `/settings/system` > Optimizer Setup.

## Рассмотрение предложений

Каждое предложение содержит:
- **Оценки сегментов**, которые привели к рекомендации
- **Конкретные изменения** полей претаргетинга (добавления, удаления, обновления)
- **Прогнозируемый эффект** на QPS, долю потерь и расходы

Вы можете:
- **Approve**: отметить предложение как одобренное
- **Apply**: применить одобренные изменения в Google
- **Reject**: отклонить предложение
- **Check apply status**: проверить, что изменения вступили в силу на стороне Google
